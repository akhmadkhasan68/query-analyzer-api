# AC: Storage File

## Context

Endpoints for file upload, retrieval, and deletion (`POST /v1/storage-file/upload`, `GET /v1/storage-file/:id`, `DELETE /v1/storage-file/:id`). Supports multiple storage drivers (Local, MinIO, Google Cloud Storage) via a factory pattern. File metadata is stored in the `storage_file` database table. All endpoints are public (no JWT required).

## Acceptance Criteria

### Upload File (`POST /v1/storage-file/upload`)

#### Happy Path

- [ ] Given a valid file attached as `multipart/form-data` (field name: `file`), when calling `POST /v1/storage-file/upload`, then the API returns `200 OK` with message "File uploaded successfully" and data containing `id`, `name`, `path`, `size` (formatted string, e.g., "1.5 MB"), `mimetype`, `fileUrl`, and `driver`.
- [ ] Given a successful upload, when the file is stored, then it is uploaded to the currently active storage driver (determined by configuration) and metadata is saved to the `storage_file` database table.
- [ ] Given the response `driver` field, when a file is uploaded, then it reflects the active storage driver enum value: `local`, `minio`, or `gcs`.

#### Validation Errors

- [ ] Given no file attached to the request, when calling `POST /v1/storage-file/upload`, then the API returns `200 OK` with message "No file uploaded" and `data: null`.
- [ ] Given a file with an invalid MIME type, when calling `POST /v1/storage-file/upload`, then the API throws an `InvalidFileMimeType` error (validated via `FileUtil.isValidFileType`).
- [ ] Given a file that exceeds the maximum allowed size (`config.storage.fileMaxSizeInBytes`), when calling `POST /v1/storage-file/upload`, then the API throws a `FileTooLarge` error.

#### Error Scenarios

- [ ] Given a valid file but the storage driver fails during upload, when calling `POST /v1/storage-file/upload`, then the API throws a `FileUpload` error and the error is logged.

#### Edge Cases

- [ ] Given the upload endpoint, when called, then it does not require JWT authentication (uses `@ExcludeGlobalGuard()`).
- [ ] Given the `size` field in the response, when returned, then it is a human-readable formatted string (e.g., "1.5 MB") generated by `FileUtil.formatFileSizeBytes()`, not a raw byte count.

---

### Retrieve File (`GET /v1/storage-file/:id`)

#### Happy Path

- [ ] Given a valid file `id`, when calling `GET /v1/storage-file/:id`, then the API returns `200 OK` with message "File retrieved successfully" and data containing `id`, `name`, `path`, `size`, `mimetype`, `fileUrl`, and `driver`.
- [ ] Given a file record with a specific `driver` value, when retrieving the file, then the `fileUrl` is regenerated using the storage driver recorded on that file's record (not necessarily the currently active driver).

#### Error Scenarios

- [ ] Given an `id` that does not exist in the `storage_file` table, when calling `GET /v1/storage-file/:id`, then the API returns `404 Not Found` (thrown by `findOneOrFail`).

#### Edge Cases

- [ ] Given the retrieve endpoint, when called, then it does not require JWT authentication (uses `@ExcludeGlobalGuard()`).

---

### Delete File (`DELETE /v1/storage-file/:id`)

#### Happy Path

- [ ] Given a valid file `id`, when calling `DELETE /v1/storage-file/:id`, then the API returns `200 OK` with message "File deleted successfully" and `data: null`.
- [ ] Given a successful delete, when the operation completes, then the file is removed from the storage driver and the metadata is removed from the database.

#### Error Scenarios

- [ ] Given an `id` that does not exist in the `storage_file` table, when calling `DELETE /v1/storage-file/:id`, then the API returns `404 Not Found` (thrown by `findOneOrFail`).
- [ ] Given a valid file `id` but the storage driver fails during deletion, when calling `DELETE /v1/storage-file/:id`, then the API throws a `FileDelete` error.

#### Edge Cases

- [ ] Given the delete endpoint, when called, then it does not require JWT authentication (uses `@ExcludeGlobalGuard()`).

---

### Multi-Driver Support

- [ ] Given the storage system, when configured, then it supports three storage drivers: `local` (local file system), `minio` (S3-compatible object storage), and `gcs` (Google Cloud Storage).
- [ ] Given a file uploaded with one driver (e.g., `minio`), when the file is later retrieved or deleted, then the system uses the driver recorded in the file's `driver` column, regardless of the currently active default driver.
- [ ] Given the `storage_file` database entity, when a file is stored, then the following columns are populated: `name` (varchar 255), `path` (varchar 255), `size` (int, raw bytes), `mimetype` (varchar 50), `driver` (varchar 50), plus BaseEntity columns.
