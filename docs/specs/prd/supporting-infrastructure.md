# Supporting Infrastructure -- PRD

## Problem

The QueryAnalyzer platform requires foundational infrastructure services for file storage (to persist AI analysis reports and user import/export files), audit trails for tracking system activity, and health monitoring to ensure system availability. Without these services, the platform cannot store generated reports, maintain operational visibility, or provide reliable uptime monitoring for dependent services.

## Solution

Provide three infrastructure modules: (1) a multi-driver file storage service supporting local filesystem, MinIO (S3-compatible), and Google Cloud Storage through a factory pattern, with file metadata persisted in the database; (2) an activity logging module (scaffolded for future implementation) for tracking user and system actions; (3) a health check module using NestJS Terminus to verify HTTP connectivity and database availability.

## Functional Requirements

### File Storage

- FR-INF-01: The system shall support file upload via `POST /v1/storage-file/upload` accepting multipart form data with a `file` field. The endpoint shall bypass the global JWT auth guard to support both authenticated and service-to-service uploads.
- FR-INF-02: The system shall validate uploaded files for allowed MIME types (using `FileUtil.isValidFileType`) and enforce a maximum file size limit (configured via `config.storage.fileMaxSizeInBytes`). Invalid file types shall throw an `InvalidFileMimeType` error; oversized files shall throw a `FileTooLarge` error.
- FR-INF-03: Upon successful upload, the system shall persist file metadata in the database as a `StorageFile` entity (including `id`, `path`, `mimetype`, `driver`) and return the file metadata along with a resolved file URL.
- FR-INF-04: The system shall support file retrieval via `GET /v1/storage-file/:id`, returning the file metadata and a resolved URL generated by the appropriate storage driver. The endpoint shall bypass the global JWT auth guard.
- FR-INF-05: The system shall support file deletion via `DELETE /v1/storage-file/:id`, which removes the file from the storage backend using the appropriate driver. The endpoint shall bypass the global JWT auth guard.
- FR-INF-06: The system shall implement a factory pattern (`StorageFactoryService`) for storage drivers, allowing the active driver to be set dynamically via `setStorageDriverService(driver)`. The factory shall provide `getStorageDriverService()` to return the currently configured default driver and support switching drivers per operation.
- FR-INF-07: Each storage driver shall implement: `uploadFile(file)` returning a `StorageFile` entity, `getFileUrl(path, mimetype?)` returning a URL string, `getFileBuffer(path)` returning a Buffer, and `deleteFile(path)` for file removal.
- FR-INF-08: The system shall support retrieving file content as a Buffer via the `getFileBuffer(id)` service method, resolving the appropriate storage driver from the file's stored `driver` field.
- FR-INF-09: The system shall support resolving file URLs via the `getFileUrl(id)` service method, used by the query analysis module to generate shareable links for AI analysis reports sent to Slack.

### Activity Logging

- FR-INF-10: The system shall provide an activity logging module with a controller endpoint at `/v1/log-activities`. The module is scaffolded with repository and service layers for future implementation.
- FR-INF-11: The RBAC system shall define `log-activity` as a resource with `view` and `import` operations, enabling permission-based access control for log data when implemented.

### Health Monitoring

- FR-INF-12: The system shall expose a public health check endpoint at `GET /v1/health` (bypassing authentication) using NestJS Terminus `@HealthCheck()` decorator.
- FR-INF-13: The health check shall verify: (a) HTTP connectivity by pinging `https://docs.nestjs.com` (indicator name: `nest-js`), and (b) database connectivity by pinging the TypeORM connection with a configurable timeout (`config.db.connectTimeoutMS`, indicator name: `database`).
- FR-INF-14: On success, the health check shall return a `"Health check passed"` message with detailed health indicator results. On failure, it shall return a `"Health check failed"` message with error details from the failing indicators.

## Out of Scope

- File versioning or revision history
- Image processing, thumbnail generation, or file transformation
- Signed URL generation with configurable expiration (handled by the storage driver)
- Activity log search, filtering, and retention policies (module is scaffolded only)
- Distributed tracing integration (OpenTelemetry, Jaeger)
- Metrics collection and Prometheus/Grafana integration
- Alerting on health check failures (to be handled by external monitoring)
- Backup and disaster recovery procedures for stored files
