# Test Scenarios: Storage File

## Preconditions

- The storage system is configured with an active storage driver (local, minio, or gcs)
- The `storage_file` database table is accessible
- File size limits are configured via `config.storage.fileMaxSizeInBytes`
- Valid file types are defined via `FileUtil.isValidFileType`
- All storage file endpoints are public (no JWT required via `@ExcludeGlobalGuard()`)

## Scenarios

### Scenario 1: Upload File Successfully

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` as `multipart/form-data` with a valid file attached (field name: `file`) | API returns `200 OK` with message "File uploaded successfully" |
| 2 | Verify response data | Contains `id`, `name`, `path`, `size` (formatted string, e.g., "1.5 MB"), `mimetype`, `fileUrl`, and `driver` |
| 3 | Verify the `driver` field | Reflects the active storage driver enum value (`local`, `minio`, or `gcs`) |
| 4 | Verify metadata is saved | File metadata is stored in the `storage_file` database table |

### Scenario 2: Upload File Without Attaching a File

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` without attaching a file | API returns `200 OK` with message "No file uploaded" and `data: null` |

### Scenario 3: Upload File with Invalid MIME Type

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` with a file that has an unsupported MIME type | API throws an `InvalidFileMimeType` error |

### Scenario 4: Upload File Exceeding Maximum Size

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` with a file that exceeds `config.storage.fileMaxSizeInBytes` | API throws a `FileTooLarge` error |

### Scenario 5: Upload File with Storage Driver Failure

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` with a valid file when the storage driver is unavailable or fails | API throws a `FileUpload` error and the error is logged |

### Scenario 6: Upload Endpoint Does Not Require JWT

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `POST /v1/storage-file/upload` without an `Authorization` header | Request is processed normally (no `401 Unauthorized`); endpoint uses `@ExcludeGlobalGuard()` |

### Scenario 7: Verify File Size Format in Response

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file of known size (e.g., 1,572,864 bytes) | API returns `200 OK` |
| 2 | Verify the `size` field in the response | Value is a human-readable formatted string (e.g., "1.5 MB") generated by `FileUtil.formatFileSizeBytes()`, not a raw byte count |

### Scenario 8: Retrieve File by ID

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file and note the returned `id` | File is uploaded and `id` is obtained |
| 2 | Send `GET /v1/storage-file/{id}` | API returns `200 OK` with message "File retrieved successfully" and data containing `id`, `name`, `path`, `size`, `mimetype`, `fileUrl`, and `driver` |

### Scenario 9: Retrieve File with Correct Driver URL

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file using a specific driver (e.g., `minio`) | File is stored with `driver: minio` |
| 2 | Change the active storage driver to a different one (e.g., `local`) | Active driver is now different |
| 3 | Send `GET /v1/storage-file/{id}` | The `fileUrl` is regenerated using the driver recorded on the file's record (`minio`), not the currently active driver |

### Scenario 10: Retrieve Non-Existent File

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `GET /v1/storage-file/{nonExistentId}` | API returns `404 Not Found` |

### Scenario 11: Retrieve Endpoint Does Not Require JWT

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `GET /v1/storage-file/{id}` without an `Authorization` header | Request is processed normally (no `401 Unauthorized`); endpoint uses `@ExcludeGlobalGuard()` |

### Scenario 12: Delete File Successfully

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file and note the returned `id` | File is uploaded |
| 2 | Send `DELETE /v1/storage-file/{id}` | API returns `200 OK` with message "File deleted successfully" and `data: null` |
| 3 | Verify the file is removed from the storage driver | File no longer exists in the storage backend |
| 4 | Verify the metadata is removed from the database | Record no longer exists in `storage_file` table |

### Scenario 13: Delete Non-Existent File

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `DELETE /v1/storage-file/{nonExistentId}` | API returns `404 Not Found` |

### Scenario 14: Delete File with Storage Driver Failure

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file and note the returned `id` | File is uploaded |
| 2 | Send `DELETE /v1/storage-file/{id}` when the storage driver fails during deletion | API throws a `FileDelete` error |

### Scenario 15: Delete Endpoint Does Not Require JWT

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Send `DELETE /v1/storage-file/{id}` without an `Authorization` header | Request is processed normally (no `401 Unauthorized`); endpoint uses `@ExcludeGlobalGuard()` |

### Scenario 16: Multi-Driver Support - Upload and Retrieve Across Drivers

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Configure storage driver to `minio` and upload a file | File is stored with `driver: minio` |
| 2 | Configure storage driver to `gcs` and upload another file | File is stored with `driver: gcs` |
| 3 | Retrieve the first file by ID | `fileUrl` is generated using the `minio` driver |
| 4 | Retrieve the second file by ID | `fileUrl` is generated using the `gcs` driver |

### Scenario 17: Verify Storage File Database Columns

| Step | Action | Expected Result |
|------|--------|-----------------|
| 1 | Upload a file successfully | File metadata is stored |
| 2 | Query the `storage_file` table directly | Record contains `name` (varchar 255), `path` (varchar 255), `size` (int, raw bytes), `mimetype` (varchar 50), `driver` (varchar 50), plus BaseEntity columns |
